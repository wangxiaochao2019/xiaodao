## 项目边界（首版冻结）

本项目首版只建立稳定、清晰、容易持续更新的个人出版系统。任何没有列入本计划的功能、内容类型、身份定位、外部服务、依赖或发布目标，均不得默认加入；需要扩展时，必须先由用户明确提出，再评估对内容迁移、永久链接、SEO、隐私和长期维护的影响。

本计划只规定未来实施方式，不代表已经创建工程、安装依赖、编写页面、执行构建、登录平台、部署网站或处理域名。首版完整禁止项见第 3 章，公开与隐私约束见第 5 章。

## 1. Summary（摘要）

本计划用于指导“晓刀｜作品、思考与长期实验”首版网站的设计、开发、验收与发布。首版建立一个稳定、清晰、容易持续更新的个人出版系统，而不是一次性堆满功能的网站。

实施采用三类状态杠杆：

- **前置杠杆**：先冻结公开边界、内容结构、配置入口、永久链接、日期时区、图片和 SEO 规则。
- **顺序杠杆**：先完成会降低后续成本的工程骨架与内容模型，再做详情页、列表页和首页。
- **返工控制**：以四个 Gate 阻断不成熟状态向后传递；本地冻结后才允许进入发布阶段。

本计划中的“必须”是首版硬约束；“建议”是默认执行方式，偏离时必须记录理由；“以后可选”不属于首版，未经用户明确提出不得实施。

## 2. Goals（目标）

首版必须达成：

- 建立以 Astro、TypeScript、Markdown 和 Content Collections 为基础的纯静态内容系统。
- 让项目、文章、短笔记、收藏和 Now 五类内容拥有清晰边界与独立 Schema。
- 让新增合规 Markdown 内容能够自动进入相应页面、列表、排序和 SEO 流程。
- 建立稳定永久链接、统一日期时区、草稿过滤、图片约束和集中站点配置。
- 以真实、已确认内容呈现晓刀的作品、思考与长期实验，不虚构经历、观点或成果。
- 完成本地浏览器、可访问性、构建、SEO、隐私和安全验收。
- 本地冻结为 v0.1.0 后，发布至 Cloudflare Workers 免费地址并完成线上验收。

## 3. Non-goals（项目边界与非目标）

首版明确禁止以下功能与技术，除非用户以后明确提出，否则不得擅自增加：

- 深色模式、站内搜索、标签详情页、分页。
- 评论、点赞、浏览量、Newsletter、访客统计、在线联系表单。
- 动态 Open Graph 图片、页面转场、Astro View Transitions。
- CMS、国际化、PWA、客户端路由。
- React、Vue、Svelte 或其他重型前端框架。
- 数据库、API、自动抓取外部内容。
- 虚假邮箱、空链接、失效外链、不存在的社交账号。
- 首版范围之外的栏目、首页模块或交互功能。

本计划不授权以下事项：

- 查询、注册或购买域名。
- 注册付费服务、开启付费功能或自动切换发布平台。
- 虚构已经完成的作品、成果、数字、版本、日期和公开地址。

## 5. Public and Privacy Boundaries（公开与隐私边界）

### 5.1 公开边界

- 只发布用户确认可以公开的内容。
- 网站公开署名始终只使用“晓刀”。
- 项目事实不足时仅标为“进行中”，不得推断或补齐成果。
- 联系方式未提供前只显示“联系方式待补充”。
- 不得创建虚假邮箱、猜测邮箱地址、联系表单、空链接、占位 mailto 链接或不存在的社交账号。
- 所有外链必须在发布前验证有效；无法验证的链接不公开。

### 5.2 禁止公开的信息

- 本地绝对路径、设备用户名、私人目录结构。
- 用户真实姓名、私人邮箱、电话号码、家庭住址、精确位置。
- 投资账户金额、账户持仓数量、敏感交易记录。
- 儿童真实姓名、儿童学校、儿童活动地点。
- API Key、Token、内部 IP、机器人密钥、自动化工具密钥、Cloudflare 密钥。
- 调试日志中的凭据、聊天记录和其他未确认可公开的信息。

### 5.3 截图发布前检查

发布任何截图前必须检查并清除：

- 文件名、Finder 侧栏、菜单栏、用户名、本地路径。
- 浏览器书签、通知内容、Token、聊天记录。
- 家庭隐私、儿童信息、账户信息和其他敏感背景。

## 6. Technology Decisions（技术决策）

### 6.1 首版技术栈

必须使用：

- Astro 当前任务执行时的稳定版本。
- TypeScript。
- 纯静态网站，配置保持 `output: static`。
- Markdown 管理内容，首版不使用 MDX。
- Astro Content Collections。
- 原生 CSS。
- Astro 官方图片能力和原生 Web 能力。
- Cloudflare Workers 承载构建后的静态产物。

不得使用数据库、后台、登录、服务端业务逻辑或重型前端框架。

### 6.2 版本与依赖约束

- 固定 Node.js 主版本，并在工程文件和维护说明中保持一致。
- 使用任务执行时的 Astro 当前稳定版本；不得使用 beta、rc、nightly 或 canary。
- 提交 `package-lock.json`，保证依赖可复现。
- 不使用未经确认的实验特性。
- 不为少量功能引入大型依赖；优先使用 Astro 官方能力和原生 Web 能力。

首版预计依赖范围：

```text
astro
@astrojs/sitemap
@astrojs/rss
wrangler（开发依赖）
```

不得引入：

- Tailwind、CSS 框架。
- React、Vue、Svelte。
- 图标组件库、动画库、搜索库。
- 日期处理大库。
- CMS SDK、Analytics SDK。

若实现必须增加其他依赖，必须先更新计划，说明原生能力为何不足、新依赖解决的问题、风险和更轻量替代方案，并取得用户确认。

### 6.3 Cloudflare 静态发布约束

- 发布目标固定为 Cloudflare Workers。
- 保持 `output: static`。
- 不安装 `@astrojs/cloudflare`。
- 不创建 KV、D1、Durable Objects、数据库、动态 Worker API 或无必要的服务端逻辑。
- 使用 Wrangler 部署 Astro 的 `dist/` 静态产物。

## 8. Content Model（内容模型）

### 8.1 共享基础字段

五类内容共享字段定义，但不强制使用完全相同的 Front Matter：

```yaml
title:
date:
description:
status:
updated:
tags:
```

规则：

- `title`、`date`、`status` 必填。
- `status` 只允许 `draft` 或 `published`。
- `updated`、`tags` 可选。
- `description` 是否必填由内容类型决定。
- Front Matter 不符合对应 Schema 时必须阻断构建。
- `draft` 不生成公开页面，也不进入列表、首页、sitemap 或 RSS。

### 8.2 Projects Schema

```yaml
title:
date:
description:
status:
updated:
tags:
featured:
cover:
coverAlt:
externalUrl:
weight:
projectStatus:
started:
version:
repositoryUrl:
demoUrl:
archiveUrl:
```

规则：

- `description`、`featured`、`weight`、`projectStatus` 必填。
- `projectStatus` 只允许：`进行中`、`已完成`、`持续更新`、`实验`、`冻结版本`。
- `started`、`version`、`repositoryUrl`、`demoUrl`、`archiveUrl`、`externalUrl`、`cover` 可选。
- 可选字段未填写时不得输出对应界面元素。
- `cover` 存在时 `coverAlt` 必填；无 `cover` 时不得渲染空图片元素。

### 8.3 Essays Schema

```yaml
title:
date:
description:
status:
updated:
tags:
featured:
cover:
coverAlt:
```

正式文章的 `description` 必填。`cover` 存在时 `coverAlt` 必填。

### 8.4 Notes Schema

```yaml
title:
date:
description:
status:
updated:
tags:
```

短笔记的 `description` 和 `tags` 可选。页面 SEO description 必须由集中规则生成：优先使用明确填写的 `description`；未填写时使用经过清理和长度控制的正文摘要。无法生成有效摘要时不得发布。

### 8.5 Collections Schema

```yaml
title:
date:
description:
status:
updated:
tags:
sourceUrl:
sourceName:
annotation:
```

每条收藏至少必须包含晓刀自己的选择理由、价值判断，以及它与长期项目、思考或问题的关联。`sourceUrl` 必须是经验证的有效外链；不得把收藏区做成只复制标题、简介和链接的外部仓库。`annotation` 必须承载原创注释，正文按模板进一步展开。

Collections 的 `description` 必填，用于列表摘要和页面 SEO；它不能替代正文中的原创注释。

### 8.6 Now Schema

Now 是独立单页内容，不套用普通文章详情页，用于记录：

- 最近在做什么。
- 最近在研究什么。
- 最近在推进哪些项目。
- 当前注意力主要放在哪里。

Now 使用共享基础字段，可省略 `tags`，但公开时必须具有有效页面 description。首版只允许一个 `published` 的当前版本；历史保留方式以后可选，不在首版建立归档页。

### 8.7 内容类型边界

**Writing**：完整文章，有清晰论点，经过整理，可以独立阅读，适合长期保存和引用。

**Notes**：短想法、临时观察、过程记录、未发展成完整文章的内容，以及项目推进中的局部心得。

**Collection**：值得保存的书、文章、工具、网站、图片、项目或其他外部材料；每条必须包含晓刀自己的选择理由、简短注释、关联思考和使用场景。

**Projects**：有持续时间、明确目标、迭代过程、可积累成果并会持续更新的长期项目。

**Now**：当前阶段状态、当前主要注意力和最近正在推进的事情。

## 9. URL and Date Rules（永久链接与日期规则）

### 9.1 永久链接

- 已发布页面的网址不得依赖中文标题自动生成。
- Markdown 文件名即永久 slug，不额外定义名为 `slug` 的 Front Matter 字段。
- 文件名统一使用小写英文和连字符。
- 不使用中文文件名作为公开 URL。
- 不使用日期作为永久链接的一部分。
- 标题以后可以修改；修改标题不得改变 URL。
- 已发布文件名原则上不得修改。
- 所有路由统一保留末尾 `/`。

示例：

```text
源文件：src/content/essays/why-this-site.md
公开地址：/writing/why-this-site/
```

如未来确需修改已发布 URL，必须新增从旧地址到新地址的重定向规则，不得直接让旧链接失效。首版只记录该原则，不实现通用重定向系统。

### 9.2 日期与时区

- 所有内容日期按 `Asia/Shanghai` 理解、排序和展示。
- 页面日期统一显示为 `YYYY.MM.DD`。
- 列表默认显示发布日期 `date`。
- `updated` 只有晚于 `date` 时才显示。
- RSS 输出标准合法日期。
- 不使用运行环境默认时区决定排序。
- 不显示虚假的阅读时长。
- 同一天内容使用稳定次级规则：先按对应日期降序，再按文件名升序。

### 9.3 排序规则

项目精选：先按 `weight` 升序，再按 `date` 降序；仍相同时按文件名升序。

最近更新：优先使用 `updated`，没有 `updated` 时使用 `date`，按时间降序；仍相同时按内容类型固定顺序和文件名升序，保证不同环境结果一致。

## 10. Image Strategy（图片与分享图策略）

### 10.1 内容图片

- 与单篇内容强绑定的图片优先放在该 Markdown 内容附近，并由内容引用。
- 全站公共图片统一放置于：

```text
public/images/
```

- 使用 Astro 官方图片能力，不引入第三方图片库。
- 内容图片必须声明准确 alt；纯装饰图片使用空 alt。
- 明确图片宽高，减少布局跳动。
- 不为填满版式而强行为每篇内容生成封面。
- 没有封面时允许纯文字卡片，不输出空图片元素。
- 有 `cover` 时 `coverAlt` 必填。

### 10.2 默认分享图

首版只制作一张全站默认分享图，尺寸为 `1200 × 630`，文字固定为：

> 晓刀  
> 作品、思考与长期实验  
> 发出信号，而不是等待被发现。

视觉要求：极简、黑白灰、一个低饱和暖色、档案线条；不得使用英文副标题、渐变、科技图形、虚假界面截图。文字必须准确。首版不实现动态文章分享图。

## 11. Site Configuration（站点配置）

所有品牌、SEO 和公开作者信息集中维护，不得散落硬编码。建议唯一配置模块为：

```text
src/config/site.ts
```

该配置必须集中包含：

- Site Title。
- 默认作者“晓刀”。
- Hero、Slogan、About。
- 导航。
- 联系方式状态与以后可补充的联系项。
- 默认 description。
- 默认 Open Graph 图。
- Canonical 基础地址。
- RSS 标题、描述和作者信息。
- 社交链接。

组件不得硬编码域名、作者、联系方式或社交账号。Canonical 只能从统一 site 配置生成。正式 Workers 地址确定前，Canonical 不得使用候选域名或猜测地址。

## 13. Page Specifications（页面规格）

### 13.1 通用页面基础

- Base Layout 统一输出 title、description、Canonical、Open Graph、语言、基础元数据和默认分享图。
- Header、Navigation、Footer 使用站点配置，不重复硬编码。
- 每页只有一个 `h1`，标题层级连续。
- 页面结构使用语义化 HTML；视觉样式使用原生 CSS。
- 所有页面支持键盘访问、可见焦点和响应式布局。

### 13.2 项目详情页：项目档案

三个首版重点项目的详情页不得只是普通 Markdown 文章。所有项目详情页统一预留：

1. 项目简介。
2. 为什么做。
3. 当前状态。
4. 开始时间。
5. 当前版本。
6. 最近更新。
7. 已完成成果。
8. 过程记录。
9. 下一步。
10. 相关链接。

未提供的字段或区块不显示，不使用“待补充链接”、`#`、空按钮、无效跳转或虚构成果。首版项目说明只使用用户已确认的信息。事实缺失时只标为“进行中”，不得自行补充数字、版本、发布日期、成果数量或公开地址。

### 13.3 文章详情页

显示标题、描述、发布日期、符合规则时的更新日期、正文和有效关联内容。文章必须可独立阅读，不显示虚假阅读时长。

### 13.4 短笔记详情页

保持轻量，显示标题、日期和正文；仅在存在有效字段时显示描述、更新日期或标签。首版标签仅作为文本信息，不链接到标签详情页。

### 13.5 收藏详情页

突出晓刀的选择理由、观察、关联思考和使用场景。来源名称与外链只在完整有效时输出，外链必须清楚标识并在发布前验证。

### 13.6 Now 与 About

Now 使用独立单页结构呈现当前状态，不套用文章详情页。About 使用固定核心文案，可补充一小段创作理念，但不得转为履历或商业服务页面。

### 13.7 列表页

列表页为：

```text

/writing/
/notes/
/collection/
```

列表页必须验证排序、草稿过滤、项目状态、日期、卡片一致性、纯文字卡片降级，以及无空链接。首版不分页、不生成标签详情页。

## 16. Accessibility（可访问性）

首版必须满足：

## 17. SEO and Feeds（SEO、RSS 与机器产物）

### 17.1 标题规则

首页标题固定为：

> 晓刀｜作品、思考与长期实验

内页标题统一为：

> 页面标题｜晓刀

每个公开页面必须有唯一 title 和有效 description。

### 17.2 索引规则

- `draft` 不生成页面，不进入 sitemap，不进入 RSS。
- 404 页面设置 `noindex`。
- 首版不生成标签详情页。
- sitemap 只包含实际公开页面。
- Canonical 只从统一 site 配置生成，不在组件中硬编码域名。
- 默认 Open Graph 图全站共用；首版不生成动态文章分享图。

### 17.3 RSS 范围

首版 RSS 只收录：

- `published` essays。
- `published` notes。

首版 RSS 不收录 projects、collections、now 或任何 `draft`。RSS 日期必须是标准合法日期，并按 `Asia/Shanghai` 语义稳定转换。

### 17.4 robots 与 404

- `robots.txt` 只指向正式地址下的 sitemap。
- 404 页面提供返回首页和主要栏目导航，不提供虚假搜索。
- sitemap、RSS、robots、Canonical 与 Open Graph 必须在正式地址写入配置并重新构建后验收。

## 18. Security and Privacy Review（安全与隐私审查）

发布前必须执行人工审查与全项目文本搜索：

- 搜索真实姓名、旧署名、私人邮箱、电话、住址、儿童信息和账户信息。
- 搜索本地绝对路径、设备用户名、私人目录结构和调试日志。
- 搜索 API Key、Token、密钥、内部 IP、Cloudflare 凭据及其他秘密。
- 检查 Markdown、配置、生成 HTML、图片文件名、图片内容和元数据。
- 检查所有外链有效且没有占位值、空目标或 `href="#"`。
- 检查公开作者始终为“晓刀”。

发现敏感信息或无法确认的外链时，必须阻断发布并回到内容修订，不得以“之后再改”为由通过 Gate。

在真正创建工程或安装依赖前，Codex 必须先查询官方资料并完成一次官方技术基线确认，确认内容包括：

完成 Astro + TypeScript、Node 主版本、Git、依赖锁定、基础目录和集中站点配置。只形成最小可运行骨架，不提前堆叠页面功能。

完成 Base Layout、Header、Navigation、Footer、SEO、Canonical、Open Graph、字体和排版变量、响应式基础与可访问性基础。

依次完成项目详情页、文章详情页、短笔记详情页、收藏详情页、Now 页面和 About 页面，先验证内容模型能够真实支撑页面。

### Phase 6：实现列表页

完成项目、文章、短笔记和收藏列表；验证排序、草稿过滤、项目状态、日期、无空链接和卡片一致性。

复用已验证的内容查询、卡片和排序规则。首页严格限制为 Hero、精选项目 3 个、最近更新 5 条、简短 About、Footer。

### Phase 10：可访问性验收

检查键盘操作、可见焦点、语义化 HTML、单一 `h1`、标题层级、图片 alt、链接可辨识、按钮语义、颜色对比、`prefers-reduced-motion`、移动导航，以及不只依赖颜色表达信息。

### Phase 11：构建和内容审查

## 21. Test Gates（验收闸门）

### Gate 1：内容模型

必须验证：

- 五类 Schema 正确，Front Matter 错误会阻断构建。
- 新增合规 Markdown 后自动生成对应页面。
- `draft` 不公开。
- 排序、日期规则、项目状态和稳定 URL 正确。
- 无空链接、占位链接或无效界面元素。

Gate 1 通过后才能进入页面视觉开发。失败时回到 Phase 2 或 Phase 3 修复。

### Gate 2：页面和视觉

必须验证：

- `375px`、`768px`、`1440px` 均无横向滚动和标题溢出。
- 导航正常、页面层级清楚、中文排版适合阅读。
- 首页结构没有扩张。
- 页面没有履历模板感、商业营销页感或夸张科技感。

失败时回到 Phase 4 至 Phase 7 修复，不得继续系统产物和发布准备。

### Gate 3：构建、SEO、安全

必须验证：

- build 成功，TypeScript 和浏览器控制台无错误。
- RSS、sitemap、robots、Canonical、Open Graph、404 正确。
- 无敏感信息、旧署名残留或失效链接。
- 草稿和未确认内容未公开。

失败时回到对应实现阶段修复。Gate 3 通过后才能进行本地冻结。

### Gate 4：发布

必须验证：

- Cloudflare 授权有效，Worker 名正确。
- Canonical 已更新为最终免费地址并重新正式构建。
- 线上页面可直达，详情页可直接刷新，手机访问正常。
- 分享图正常，RSS 和 sitemap 在线可访问。

Gate 4 全部通过后才可声明首版发布完成。

具体部署命令必须在实施时依据所用 Wrangler 稳定版本的官方静态资源配置确认，确认后记录在维护文档中，不在本计划中猜测过时参数。
